# 科研文献智能解析与知识服务系统 — 任务分工文档

> 使用说明：请三人各自先看一遍全文，标注自己不确定的任务，然后开会逐条过。

## 角色总览

| 角色 | 代号 | 一句话职责 |
|------|------|-----------|
| 后端开发 + 数据库 | **A** | 管系统骨架：用户系统、数据库、所有 API 接口、文件存储 |
| AI 管道开发 | **B** | 管智能能力：PDF 解析、LLM 抽取、向量化、检索引擎、问答生成、报告生成 |
| 前端开发 | **C** | 管用户界面：页面框架、文献列表、论文详情、设置页、问答交互、报告预览 |

## 一、A 的任务（后端开发 + 数据库）

### 1.1 项目初始化（第 1 天）
- [ ] 创建项目目录结构（backend/app/、backend/data/、backend/prompts/）
- [ ] 编写 `requirements.txt`、`config.py`
- [ ] FastAPI 入口文件 `main.py` + CORS 配置
- [ ] uvicorn 启动脚本，确认 `localhost:8000/docs` 可访问

### 1.2 数据库设计与实现（第 1-2 天）
- [ ] 设计建表 SQL / SQLAlchemy 模型，至少包含以下表：

| 表名 | 核心字段 |
|------|---------|
| users | id, username, password_hash, created_at, is_anonymous |
| papers | id, user_id(FK), title, authors(json), file_name, file_size, file_path, upload_time, parse_status, field, tags(json), read_status |
| paper_structured_info | id, paper_id(FK), research_background, research_questions, method_flow, model_algorithm, dataset_info, evaluation_metrics(json), experiment_results, innovations, limitations, future_work, figures_tables(json), references(json) |
| chunks | id, paper_id(FK), section_title, page_number, paragraph_index, content |
| conversations | id, paper_id(FK), user_id(FK), created_at |
| messages | id, conversation_id(FK), role(enum), content, sources(json), created_at |
| reports | id, paper_id(FK), user_id(FK), report_type(enum), content, generated_at |

- [ ] 编写数据库初始化脚本（建表 + 迁移）
- [ ] 确认 SQLite 文件路径配置

### 1.3 用户系统（第 1-2 天）
- [ ] POST `/api/auth/anonymous` — 生成匿名 Session ID，创建临时用户记录
- [ ] POST `/api/auth/register` — 注册（用户名 + 密码，密码 hash），注册后自动将匿名数据迁移至新账户
- [ ] POST `/api/auth/login` — 登录校验，返回 token，同时返回 `has_anonymous_data: bool` 供前端判断是否弹窗
- [ ] POST `/api/auth/merge-anonymous` — 将当前 Session 的匿名数据迁移至登录账户
- [ ] 中间件：从请求头解析身份（匿名 Session 或登录 Token），注入 request 上下文
- [ ] GET `/api/user/config` — 获取用户配置
- [ ] PUT `/api/user/config` — 更新 API Key / 模型选择

### 1.4 论文管理接口（第 2-3 天）
- [ ] POST `/api/papers/upload` — 接收 PDF 文件，校验（扩展名、大小、PyMuPDF 快速检测可读性），保存文件，写入 papers 表，返回 paper_id。**状态初始为"待解析"，不在此接口内触发解析。**
- [ ] GET `/api/papers` — 论文列表，支持 `?keyword=&tag=&status=&page=&size=` 筛选和分页
- [ ] GET `/api/papers/{paper_id}` — 论文详情，返回 papers 表 + paper_structured_info 表联合查询
- [ ] PUT `/api/papers/{paper_id}` — 更新论文信息（标签、阅读状态等）
- [ ] DELETE `/api/papers/{paper_id}` — 删除论文记录 + 关联的结构化信息、对话、报告、本地文件和 FAISS 索引文件
- [ ] POST `/api/papers/{paper_id}/reparse` — 重新触发解析

### 1.5 触发异步解析（第 3 天）
- [ ] 上传完成后，在 BackgroundTasks 中调用 B 提供的解析函数
- [ ] 解析开始 → 更新 papers 表 status = "parsing"
- [ ] 解析成功 → 更新 status = "completed"，写入 paper_structured_info 表
- [ ] 解析失败 → 更新 status = "failed"，写入错误原因字段

### 1.6 问答接口（第 3-4 天）
- [ ] POST `/api/qa/{paper_id}` — 接收 `{question, conversation_id?}`，调用 B 的问答函数，返回 `{answer, sources, conversation_id}`
- [ ] GET `/api/qa/{paper_id}/history` — 返回该论文的对话历史

### 1.7 报告接口（第 4-5 天）
- [ ] POST `/api/reports/{paper_id}` — 接收 `{report_type}`，调用 B 的报告生成函数，返回 `{content, format}`
- [ ] 报告生成后存入 reports 表

### 1.8 联调与收尾（第 5-10 天）
- [ ] 配合 C 联调所有接口，修正字段不一致
- [ ] 配合 B 调通异步解析的状态流转
- [ ] 异常处理完善（统一错误码、日志记录）

## 二、B 的任务（AI 管道开发）

### 2.1 环境准备（第 1 天）
- [ ] 下载 BGE 模型 `BAAI/bge-large-zh-v1.5`
- [ ] 安装 PyMuPDF、pdfplumber、sentence-transformers、faiss-cpu、langchain
- [ ] 测试：对一篇示例 PDF 跑通 PyMuPDF 文本提取
- [ ] 获取 LLM API Key（DeepSeek 或 Qwen），测试一个简单调用

### 2.2 PDF 解析函数（第 1-3 天）
- [ ] 函数 `parse_pdf(file_path: str) -> dict`
  - 输入：PDF 文件路径
  - 输出：`{ "full_text": "全文文本（按章节组织，带页码标注）", "sections": [{"title": "引言", "content": "...", "page_start": 1, "page_end": 3}, ...], "figures_tables": [{"number": "Fig.1", "title": "...", "page": 4}, ...], "references_raw": "参考文献原文" }`
- [ ] 处理双栏排版（PyMuPDF 的文本排序）
- [ ] 章节结构识别（基于正则匹配标题模式，如 `1.` `1.1` `Introduction` 等）
- [ ] 异常处理：PDF 损坏、加密、扫描件（返回明确错误信息，不崩溃）

### 2.3 结构化信息抽取函数（第 2-3 天）
- [ ] 函数 `extract_structured_info(full_text: str, sections: list) -> dict`
  - 输入：parse_pdf 的文本输出
  - 输出：包含全部结构化字段的 dict（标题、作者、方法、创新点等，参考 A 的 paper_structured_info 表字段）
- [ ] 编写抽取 Prompt 模板（放入 `backend/prompts/extraction.txt`）
- [ ] Prompt 强制要求输出 JSON 格式
- [ ] 容错处理：LLM 输出非标准 JSON 时做正则兜底解析

### 2.4 知识库构建函数（第 3-4 天）
- [ ] 函数 `build_knowledge_base(paper_id: str, sections: list) -> bool`
  - 输入：paper_id + parse_pdf 的 sections 输出
  - 步骤：
    1. 文本分块（按语义段落，500-800 字符/块，相邻块重叠 10-20%）
    2. 每块标注元数据（paper_id、section_title、page_number、paragraph_index）
    3. BGE 向量化所有分块
    4. 创建 FAISS 索引文件，命名规则 `data/vectors/{paper_id}.index`
    5. 分块内容 + 元数据写入 chunks 表（由 A 的数据库提供，或 B 直接调用 A 的数据访问层）
  - 删除索引函数 `delete_index(paper_id: str)`

### 2.5 检索函数（第 4 天）
- [ ] 函数 `search_chunks(paper_id: str, query: str, k: int = 5) -> list[dict]`
  - 输入：paper_id + 用户问题文本 + 返回条数
  - 输出：`[{"content": "片段文本", "section_title": "4.1实验设置", "page": 4, "score": 0.89}, ...]`
  - 步骤：BGE 向量化 query → 加载 FAISS 索引 → 相似度搜索 → 返回 Top-K

### 2.6 问答生成函数（第 4-6 天）
- [ ] 函数 `generate_answer(paper_id: str, question: str, conversation_history: list | None) -> dict`
  - 输入：paper_id + 问题 + 对话历史（可选，多轮时传入）
  - 输出：`{ "answer": "生成的回答", "sources": [{"page": 4, "section": "4.1", "snippet": "原文摘录"}], "conversation_id": "xxx" }`
- [ ] 多轮对话支持：
  - 参数 `conversation_history` 非空时，先调用 LLM 改写问题（补全指代）
  - 改写后的 query 再去做检索
  - Prompt 中附带对话历史上下文
- [ ] 编写问答 Prompt 模板（`backend/prompts/qa_system.txt`）
- [ ] 异常处理：LLM 超时重试 1 次、检索结果为空时的回答策略

### 2.7 报告生成函数（第 5-7 天）
- [ ] 函数 `generate_report(paper_id: str, report_type: str) -> str`
  - 输入：paper_id + 报告类型（"quick" / "method" / "experiment"）
  - 输出：Markdown 格式的报告全文
- [ ] 三种报告模板（`backend/prompts/report_templates/`）
- [ ] 不同报告类型检索侧重点不同：速读→摘要/结论，方法→方法章节，实验→实验章节

### 2.8 与 A 的对接（贯穿全程）
- [ ] 确认 A 的数据库表结构，了解 chunks、paper_structured_info 表的字段
- [ ] 确认文件路径约定：PDF 存哪里、FAISS 索引导哪里
- [ ] 确认函数返回值结构，确保 A 的 API 层可以直接使用

### 2.9 调优（第 7-10 天）
- [ ] 用 3-5 篇不同类型论文测试解析质量，调整分块策略
- [ ] Prompt 调优：抽取准确率、问答质量、报告可读性
- [ ] 异常场景测试：空论文、超长论文、非英文/中文论文

## 三、C 的任务（前端开发）

### 3.1 项目初始化（第 1 天）
- [ ] 选型确定（Vue 3 或 Streamlit），如果选 Vue 需要额外配 Vite + Router + 组件库
- [ ] 项目目录结构搭建
- [ ] 全局样式（配色、字体、布局）、通用组件（Header、Sidebar、Loading、ErrorPrompt）
- [ ] 封装 HTTP 请求模块（axios / fetch），统一处理错误码和提示

### 3.2 文献列表页（第 1-3 天）
- [ ] 页面路由：`/` 或 `/papers`
- [ ] 布局：顶部搜索栏 + 筛选器（标签下拉、状态筛选），下方论文卡片/列表
- [ ] 搜索：关键词搜索（调 A 的 GET `/api/papers?keyword=`）
- [ ] 筛选：按标签、按解析状态筛选
- [ ] 列表项内容：标题、作者、上传时间、文件大小、解析状态标签（不同颜色）、用户标签
- [ ] 空状态：首次使用显示引导提示
- [ ] 加载状态：列表加载中 skeleton 占位
- [ ] 分页：无限滚动或分页按钮

### 3.3 上传功能（第 2-3 天）
- [ ] 【上传论文】按钮 → 弹出文件选择器
- [ ] 前端校验：文件扩展名（.pdf）、文件大小（≤50MB）
- [ ] 上传进度条
- [ ] 上传成功后：列表顶部插入新记录，状态 ="待解析" → 立即变为"解析中"
- [ ] 解析状态轮询：每 3 秒调 GET `/api/papers?status=parsing` 刷新状态
- [ ] 解析完成：卡片状态标签自动更新为"已完成"，绿色
- [ ] 解析失败：卡片状态标签显示"失败"（红色），悬浮显示错误原因，提供【重新解析】按钮

### 3.4 设置页（第 3 天）
- [ ] 页面路由：`/settings`
- [ ] API Key 输入框（密码模式显示）
- [ ] 模型选择下拉框
- [ ] 【测试连接】按钮：调用 A 的 API 测试 Key 有效性
- [ ] 保存后即时生效

### 3.5 论文详情页（第 3-7 天，核心页面，工作量最大）
- [ ] 页面路由：`/papers/{paper_id}`
- [ ] **左右分栏布局**（用 CSS Grid 或 Flexbox）：
  - 左栏（~60% 宽度）：原文文本展示 + 章节导航
  - 右栏（~40% 宽度）：上半问答区 + 下半结构化信息卡片

- [ ] **左栏 — 原文展示：**
  - 从 A 的 GET `/api/papers/{paper_id}` 获取全文文本
  - 按章节折叠/展开
  - 页码标注
  - 文本支持选中复制

- [ ] **右栏上半 — 问答区：**
  - 对话气泡样式（用户问题靠右，回答靠左）
  - 输入框 + 发送按钮
  - 发送后显示"检索中…"→"生成中…"
  - 回答中来源引用显示为可点击标签（`[第4页，4.1节]`）
  - 多轮对话：新问题追加到对话列表，保留上下文
  - 空状态提示："输入问题开始与论文对话"

- [ ] **右栏下半 — 结构化信息卡片：**
  - 折叠式卡片组：标题/作者 | 研究背景 | 方法流程 | 实验数据 | 创新点 | 局限性
  - 默认展开基础信息，其余折叠
  - 从 GET `/api/papers/{paper_id}` 获取数据

- [ ] **报告生成入口：**
  - 右栏顶部或浮动按钮：【生成报告】
  - 点击弹出类型选择：速读报告 / 方法总结 / 实验总结
  - 生成中显示进度动画
  - 报告在左侧或弹窗中以 Markdown 渲染预览
  - 提供【复制】和【导出】按钮

### 3.6 通用交互细节（第 5-7 天）
- [ ] 错误统一处理：网络异常、API 返回错误码、LLM 调用失败
- [ ] 空状态：文献列表空、问答区空、报告未生成
- [ ] 加载状态：列表加载、上传进度、解析中、问答生成中、报告生成中
- [ ] 用户登录/注册弹窗：点击导航栏按钮弹出
- [ ] 匿名数据合并弹窗：登录成功后若 A 返回 `has_anonymous_data: true`，弹窗三选项（合并 / 暂不 / 放弃）

### 3.7 联调（第 7-10 天）
- [ ] 与 A 逐接口联调：字段对齐、错误码处理、加载状态
- [ ] 界面适配：不同浏览器、窗口大小下的布局表现
- [ ] 交互细节打磨：动画流畅度、按钮反馈、错误恢复

## 四、三人之间的交接清单

### A → B 的交接（第 2 天，30 分钟）

A 需要确认给 B：

| 事项 | 具体内容 | B 需要确认 |
|------|---------|-----------|
| 数据库访问方式 | B 直接调 A 的数据库模块，还是 B 只返回数据让 A 自己存？**建议：B 返回 dict，A 负责写入数据库。** | B 不需要关心数据库细节 |
| PDF 文件路径 | A 上传后存到 `data/uploads/{paper_id}.pdf` | B 的 parse_pdf 从该路径读取 |
| FAISS 索引路径 | `data/vectors/{paper_id}.index` | B 的 build_kb 写到该路径 |
| 解析状态更新 | B 的解析函数不要直接改数据库，通过返回值（或回调）告诉 A 结果，A 来改 papers 表 | B 返回 `{"success": bool, "error": str}` |
| 结构化信息字段 | A 给出 paper_structured_info 表的字段列表 | B 的 extract_structured_info 返回 dict 字段名对齐 |

### A → C 的交接（第 2 天，30 分钟）

A 需要给 C：

| 事项 | 具体内容 |
|------|---------|
| API 文档 | 一份 Markdown 或 Swagger 文档，含所有接口的路径、方法、请求体、返回体、错误码 |
| Mock 数据 | 第 2 天先返回假数据，让 C 能开始写前端（如 GET /papers 返回两条写死的论文记录） |
| 接口地址 | 后端跑在 `localhost:8000`，前端配置的 baseURL |

C 需要反馈给 A：
- 每个接口返回的字段是否够用（比如列表页需要的字段在返回体里有没有）
- 多轮问答时，conversation_id 是后端生成还是前端传

### B → C（通过 A 中转，第 4 天，15 分钟）

理论上 C 不需要直接了解 B 的实现。但有一件事需要对齐：

| 事项 | 需要确认 |
|------|---------|
| 问答响应中 sources 的格式 | C 需要在界面上显示来源，需要 `page_number` 和 `snippet`。B 输出的 sources 必须包含这两个字段。这个约定的传递人是 A（A 在写 `/api/qa` 时要求 B 按此格式返回）。 |

## 六、讨论会建议（发给组员前先看，开会时逐条过）

请每位组员在开会前：
1. 通读自己的任务清单，标记"不确定能不能做"的项
2. 通读交接清单，看自己需要别人给的东西是否明确
3. 记下任何疑问，开会时提出来

开会讨论三个议题：
1. 自己的任务哪些有困难（技术能力、时间不够）
2. 交接清单有没有遗漏（我需要的东西别人没列）
3. 时间线是否合理（第一周的依赖关系能不能撑住）
