# 创新功能需求文档（v1.0）

在现有「单篇论文 RAG 系统」基础上，新增四个功能：
**A 跨论文对比** · **B 混合检索+重排** · **C 首页推荐（展示位+标签匹配）** · **D 知识图谱**。

> 原则：四个功能**基本不动数据库主结构**（仅 C 加极小字段），以「新增接口 + AI 逻辑 + 前端页面」为主，风险可控。

---

## 功能 A：跨论文对比（三视图）

### 目标
用户勾选 2~5 篇已解析论文，自动生成横向对比表 + 一段综述小结，支持三种对比视角切换。

### 对比数据来源
全部复用已抽取的 `PaperStructuredInfo`，无需重新解析：

| 维度 | 字段 |
|------|------|
| 研究问题 | research_questions |
| 方法/模型 | method_flow / model_algorithm |
| 数据集 | dataset_info |
| 评估指标 | evaluation_metrics |
| 创新点 | innovations |
| 局限 | limitations |

### 三种对比视角（同一套数据，换模板与展示行）
- **综合对比**（默认）：全维度表格
- **方法对比**：聚焦方法/模型，附方法差异分析
- **实验对比**：聚焦数据集/指标/结果，找「同 benchmark 谁更优」

### 接口（新增）
- `POST /api/reports/comparison`
  - body：`{ "paper_ids": [...], "view": "overall|method|experiment" }`
  - 返回：SSE 流式 markdown（综述部分），复用现有 `generate_report_stream` 流式模式
- 对比表格数据可随首帧一次性返回（来自 DB，无需等 LLM）

### 接入点
- `ai/report_generator.py`：新增 `report_type="comparison"` + 三套视角 prompt
- `app/api/reports.py`：新增对比路由
- 前端：列表页「多选模式」→ 选中 ≤5 篇 → 底部「对比」按钮 → 对比结果页

### 界面设计
```
┌──────────────────────────────────────────────┐
│ 已选 3 篇   [综合对比 ▾]        [开始对比]     │
├──────────────────────────────────────────────┤
│ 维度      │ 论文A   │ 论文B   │ 论文C         │  ← 首列冻结，横向滚动
│ 研究问题  │ ...     │ ...     │ ...           │
│ 方法      │ ...     │ ...     │ ...           │
│ 数据集    │ ...     │ ...     │ ...           │
│ 指标/结果 │ ...     │ ...     │ ...           │
├──────────────────────────────────────────────┤
│ 📝 综述小结（LLM 流式生成的横向评述）         │
└──────────────────────────────────────────────┘
```

### 等待体验（关键）
1. 点「开始对比」→ **对比表格立即渲染**（数据来自 DB，0 等待）
2. 下方综述区骨架屏 + SSE 逐字流式输出
→ 用户几乎无感等待。

### 验收
- 选 3 篇能即时出表格；综述在 30s 内流式完成；每列标注来源论文；三视角可切换。

#### 数据来源分两层（关键设计）

对比结果拆成成本完全不同的两部分：

| 部分 | 来源 | 成本 | 时机 |
|------|------|------|------|
| **对比表格** | `PaperStructuredInfo` 结构化字段（DB 直读） | ~0 | SSE 首帧 `{type:"table"}`，立即渲染 |
| **综述小结** | LLM 流式生成 | 高 | 表格后逐字 `{type:"delta"}` |

> 表格**不落库、不缓存**：它由结构化字段实时组装，一旦某篇重新解析，缓存会失真。前端仅按视角做**会话内缓存**（切回已生成视角秒出）。

#### 综述做了检索增强（档 2，突破抽取天花板）

结构化字段是解析时 LLM **只读前 5000 字**（见 `info_extractor.py:_build_extraction_prompt`）抽出的压缩摘要，实验结果/局限等后半部内容常缺失。因此综述**不只用结构化字段**：

- `_retrieve_evidence()` 对**每篇论文的整篇 chunks**（chunks 覆盖全文，不受 5000 字限制）按视角检索最相关原文片段（复用 B 的 `search_chunks` + 各视角检索词 `COMPARISON_QUERIES`），喂给 LLM 作为"对比以此为准"的证据。
- **Token 封顶**：证据总预算 `COMPARISON_MAX_CONTEXT=12000` 字，**按篇数均摊**（5 篇则每篇 2400），论文越多每篇越少、总量恒定——不靠拆分多次调用，保留单次调用的跨篇同框对比质量与流式体验。
- **单字段截断** `COMPARISON_FIELD_MAXLEN=220`，防个别超长字段挤爆上下文。
- **降级保护**：无 FAISS 索引 / 无分块 / `jieba`·`rank_bm25` 缺失时，`_retrieve_evidence` 返回空串，综述自动退回纯结构化字段（即档 1）。

#### 接口实现细节（与需求的差异）

- 路由 `POST /api/reports/comparison` 必须**注册在 `POST /{paper_id}` 之前**，否则 `comparison` 会被当成 `paper_id` 匹配。
- body：`{ "paper_ids": [...], "view": "overall|method|experiment" }`（去重保序，校验 2~5 篇、均属当前用户、均 `completed`）。
- SSE 事件：`{type:"table", table:{view,view_title,papers,rows}}` → 多个 `{type:"delta", content}` → `{type:"done", view}`；错误 `{type:"error", message}`。
- **不落库**：需求原文标注"可选复用 `reports`"，但 `reports.paper_id` 是单外键、装不下跨多篇对比，故本期不持久化。若后续要"对比历史"，新建 `comparisons` 表（`user_id` + `sorted(paper_ids)` 哈希做缓存键 + `view` + `summary`），并注意重新解析导致的失效。

#### 前端体验

- 首列冻结、横向滚动的对比表（首帧秒渲染）+ 综述骨架屏 + `marked` 流式渲染。
- 三视角切换、**视角会话缓存**、复制综述、导出整份为 Markdown、失败重试、长单元格（>120 字）折叠展开。

## 功能 B：混合检索 + 重排

### 目标
将纯向量检索（FAISS L2）升级为 **BM25 关键词 + 向量** 混合检索，提升问答/报告的引用准确度。

### 现状
`ai/knowledge_base.py:search_chunks()` 仅用 FAISS 向量。

### 本期实现（档位 1，无需部署新模型）
1. 引入 `rank_bm25`（纯 Python 库，非模型，不吃显存）对同批 chunks 做关键词打分
2. 用 **RRF（Reciprocal Rank Fusion）** 融合「向量排名」与「BM25 排名」
3. 向量部分保持 bge-large-zh 不变

### 待办（档位 2，本期不做）
- [ ] cross-encoder 重排（`bge-reranker-base`，约 1GB，需额外部署）——留开关 `use_rerank`，后续视资源加

### 配置
`ai/config.py` 的 `SEARCH_CONFIG` 增加：`use_hybrid`（默认 true）、`use_rerank`（默认 false）、`rerank_top_n`

### 兼容性
`search_chunks` 签名不变（`paper_id, query, k, chunks_data`），QA 与报告调用方**无需改动**。

### 验收
- 准备 5~10 个测试问答，混合检索命中率 ≥ 纯向量
- 提供开关，可做「开/关」前后对比（作为实验论据）

## 功能 C：首页推荐（展示位 + 标签匹配）

### 目标
首页顶部新增「精选推荐」区，界面美观。由两部分组成：
1. **管理员展示位（主）**：管理员后台手动挑选论文设为推荐，前端固定展示
2. **标签匹配轻量推荐（点缀）**：按 `field`/`tags` 重合度 + 未读优先，给当前用户排序推荐

### 标签匹配逻辑（零模型）
- 取用户近期论文的 `tags`/`field`
- 与库中其他论文求标签交集，交集越多越靠前
- `read_status=unread` 优先，同 `field` 加权
- 纯排序逻辑，几十行代码，无额外成本

### 数据模型（极小改动）
`papers` 表新增两列（Alembic 迁移）：
- `is_recommended` BOOLEAN 默认 false
- `recommend_order` INT 可空（排序用）

### 接口（新增）
- `GET /api/papers/recommendations?limit=6`
  - 返回管理员推荐位 + 标签匹配结果，每条带 `reason`（推荐理由）
- 后台：管理端增加「设为推荐 / 取消 / 排序」操作（复用 `admin.py` + `admin_page.html`）

### 界面（重点：好看）
```
🔥 精选推荐
┌─────────┐ ┌─────────┐ ┌─────────┐
│ 领域色标 │ │         │ │         │   ← 渐变卡片 + hover 微动效
│ 标题     │ │         │ │         │
│ 作者·领域│ │         │ │         │
│ 推荐理由 │ │         │ │         │
│ [去阅读] │ │         │ │         │
└─────────┘ └─────────┘ └─────────┘
```

### 验收
- 有推荐时展示 ≤6 张卡片，每张含一句推荐理由；空库时该区不显示
- 管理员可增删/排序推荐位，前端实时反映

#### 推荐逻辑（两层）

1. **管理员推荐位（主）**：`is_recommended=true` 跨用户，按 `recommend_order` 升序（NULL 排最后）；`reason="管理员精选"`，`recommend_source="admin"`。匿名/登录都可见。
2. **标签匹配（点缀，补足到 limit）**：取当前用户近期 completed 论文聚合 `field`/`tags`，对其**自己**的其它 completed 论文打分（tags 交集 ×10 + 同 field +5 + unread +2），`score>0` 才推，降序；`reason` 动态生成，`recommend_source="tag"`。

#### 跨用户只读放行（推荐论文可点开阅读）

管理员推荐位是**跨用户**的，但 `GET /api/papers/{paper_id}` 原本带 `user_id` 归属校验，导致用户点开**他人的**推荐论文会 404。已改：`get_paper` 先按 `paper_id` 查，`本人 OR is_recommended=true` 才放行，否则 404。**仅此只读接口放行**，`update`/`delete`/`reparse` 等写操作的归属校验不变。已验证陌生 session 能打开推荐论文、非推荐他人论文仍 404。

#### 关键坑（已修复）

- **NULL 兼容**：迁移新增列后，历史行 `is_recommended` 为 `NULL`（ORM `default=False` 只对新插入生效）。SQL 中 `is_recommended == False` **不匹配 NULL 行**，导致标签匹配候选几乎全被漏掉。候选过滤必须用 `Paper.is_recommended.isnot(True)`（含 False 与 NULL）。
- **路由顺序**：`GET /recommendations` 必须注册在 `GET /{paper_id}` 之前（与功能 A 的 comparison 同一个坑）。
- **axios 响应拦截器**：`frontend/src/api/index.js` 的响应拦截器已 `return response.data`，故 `papersAPI.xxx()` 直接返回业务体（如 `{items}`），调用方**不要**再解构 `const { data } = ...`。首次接入推荐时误写了一层解构，导致 `data.items` 抛错被 catch 吞掉、推荐区静默不显示。正确写法：`const res = await papersAPI.recommendations(); res.items`。

## 功能 D：知识图谱（vis-network）

### 目标
可视化用户文献库中论文之间的关系：引用、共用数据集、共用方法，支持拖拽/点击/展开。

### 数据来源
- 节点：论文 / 数据集(`dataset_info`) / 方法(`model_algorithm`)
- 边：
  - 论文 —引用→ 论文（`_split_references` 参考文献 + 标题匹配）
  - 论文 —使用→ 数据集 / 方法

### 匹配策略：AI 负责语义、规则兜底
- 数据集/方法节点：沿用现有 LLM 抽取结果（不额外调用）
- 引用匹配：**先规则模糊匹配**，拿不准的模糊 case 才交 AI 判断（控制 token 成本）
- 名称归一化：数据集/方法做别名/大小写/版本号归一（可用小型别名表 + AI 辅助）

### 防错设计（宁可少连，不可错连）
1. **置信度阈值**：低于阈值不连边（漏 > 错）
2. **不凭空造节点**：匹配不上库内论文的参考文献不生成节点
3. **可溯源**：点边可查看其依据（原始参考文献/字段）
4. **可手动纠错**：节点/边支持删除、编辑（承认 AI 会错 + 给用户纠错入口）
5. **范围收敛**：引用质量不佳时，优先展示更可靠的「共用数据集/方法」聚类关系

### 接口（新增）
- `GET /api/graph` → `{ nodes: [{id,label,type}], edges: [{source,target,relation,confidence,evidence}] }`

### 前端
- 选型：**vis-network**（网络图交互开箱即用，视觉贴近「知识图谱」）
- 支持拖拽、缩放、点击节点跳论文详情、点击边看溯源、手动删/改

### 验收
- ≥3 篇论文时渲染可交互图谱；错连率通过阈值控制在可接受范围；支持人工纠错

- **可溯源 + 可纠错**：每条边带 `evidence`（引文原文 / 字段片段）+ `confidence`；前端点边看溯源，删节点/边**仅会话内生效**（刷新恢复，不落库）。
- **降级保护**：无 API Key 整图仍渲染（规则实体 + 规则高置信引用），返回 `llm_used` 标记；单篇实体抽取失败不影响整图。
